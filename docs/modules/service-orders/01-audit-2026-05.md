## Critical Issues (must fix)

### 1. тЭМ Three API endpoint stubs тАФ return 404 in production

The UI actively calls these endpoints but no `route.ts` exists:

| Endpoint | Called from | Expected behavior |
|----------|-------------|-------------------|
| `POST /api/service-orders/[id]/nps` | `page.tsx:1302` | Save customer NPS score (1тАУ5) + comment on CLOSED orders |
| `POST /api/service-orders/[id]/duplicate` | `page.tsx:1362` | Clone OS to a new DRAFT with same client/scope |
| `POST /api/service-orders/[id]/request-reimbursement` | `page.tsx:1385` | Create reimbursement Expense from a RECEIPT attachment |

Every button press results in a **404** silently failing.

### 2. тЭМ Schema gap тАФ 4 fields referenced in UI but missing from Prisma schema

```
ServiceOrder.npsScore         (Int?)
ServiceOrder.npsComment       (String? Text)
ServiceOrder.npsRespondedAt   (DateTime?)
ServiceOrder.slaDeadline      (DateTime?)
```

These are declared in the UI's `ServiceOrderData` interface and used for rendering. Without them in the schema, NPS is never stored and SLA deadline is always null.

## Fixes Applied

### Fix 1 тАФ Schema: Added 4 fields to `ServiceOrder`

```prisma
npsScore       Int?      @map("nps_score")
npsComment     String?   @db.Text @map("nps_comment")
npsRespondedAt DateTime? @map("nps_responded_at")
slaDeadline    DateTime? @map("sla_deadline")
```

Applied via `npx prisma db push`.

### Fix 2 тАФ Implemented `POST /api/service-orders/[id]/nps`

- Auth: `requireUser` + `can('service-orders', 'update')`  
- Input: `{ score: 1тАУ5, comment?: string }`  
- Business rules: order must be `CLOSED`; one-time only (409 if already scored)  
- Updates: `npsScore`, `npsComment`, `npsRespondedAt = now()`

### Fix 3 тАФ Implemented `POST /api/service-orders/[id]/duplicate`

- Auth: `requireUser` + `can('service-orders', 'create')`  
- Copies: client, title, description, address, materialSupply, schedule, rates, scope items  
- New state: `DRAFT`, new `ticketNumber`, `createdById = user.id`  
- Returns: `{ newServiceOrder: { id, ticketNumber } }`

### Fix 4 тАФ Implemented `POST /api/service-orders/[id]/request-reimbursement`

- Auth: `requireUser` + `can('service-orders', 'update')`  
- Input: `{ attachmentId: number }`  
- Creates `Expense` record: type `OPERACIONAL`, category lookup or create, `status = AGUARDANDO_APROVACAO`, `requerAprovacao = true`  
- Attachment must be `RECEIPT` type and belong to this service order

### Fix 5 тАФ `empresaId` antipattern

Replaced `Number((user as any).empresaId) || 1` with `const EMPRESA_ID = 1` in:
- `bulk-confirm/route.ts`
- `materials/[materialId]/route.ts`

## Recommendations

1. **SLA deadline setter** тАФ `slaDeadline` is now in the schema but nothing sets it yet. Consider auto-setting it on OS creation (e.g., SCHEDULED + 48h, IN_PROGRESS + 24h). For now, it can be set manually via the PUT `/api/service-orders/[id]` route.

2. **NPS UI update** тАФ After the route is live, the GET endpoint for `[id]` should now return `npsScore`, `npsComment`, `npsRespondedAt`. Verify the `select` in `[id]/route.ts` includes those fields.

3. **approve/reject-purchase RBAC** тАФ These routes are secure but use inline role arrays. A future pass could refactor to `can()` for consistency.
